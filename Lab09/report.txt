4. List the exact line numbers in your bank.c source code that Helgrind flagged. Explain why these specific lines are not atomic (break them down into the 3-step CPU cycle).
    Lines 24-26. These are not atomic, the cycle is as follows.
        loads bank[from_idx].balance into a reg
        subtract amount from that reg
        store that reg value back in memory

        same process for line 26, but with addition instead of subtraction

        for line 24, it loads bank[from_idx].balance into a reg
        compares with amount and sets flags
        decides whether to branch or not

    These are not atomic operations because values can change after reads and before writes.
    Threads can interleave at the time these steps are carried out, and they will write the wrong values.

    real time: 0m 0.385s
6. Explain why this version is mathematically sound but architecturally “sequential.” If you have an 8-core machine, how many cores are actually doing useful work at any given moment?
    This is mathematically sound because the lock forces only one thread to execute this code at a time, preventing interleaving and the race condition.
    It is architecturally sequential because only one thread can work here at a time while the others do nothing. This slows execution of the program heavily. With 8 cores, only one core is working at a time once it gets to this code, like a bottle neck.


Part 4: The Deadly Embrace
7. Describe the behavior of your program as you worked through implementing fine-grained locks (one per account). (This is a bit free-form. Be as specific as you can. Describe your successes, describe your challenges and failures!)
    I did not have much difficulty in implementing the fine grained lock. These are the steps I took:
        1. move the lock declaration from the global variable into the struct
        2. in all mutex functions call them as a &bank[account].lock
        3. move the lock initialization into the for loop in main, so it gets init for each account
        4. lock both accounts, unlock both accounts on either side of the transfer() logic
    With this lock implementation, the program works sometimes. When it works there is no leak. When it doesn't work, it just doesn't finish. This is a deadlock.
8. Explain the Circular Wait condition. Why does Thread A holding Account 1 (waiting for 2) and Thread B holding Account 2 (waiting for 1) result in a deadlock?.
    They are waiting on each other. A waits on B, B waits on A. Neither moves first and proceeds, causing the deadlock. This is because this mutex implementation has no enforced order or hierarchy which causes a specific thread to proceed in case of this deadlock.

10. Provide a informal proof of why Total Ordering (Lower-Index First) makes a “Circular Wait” mathematically impossible.
    Given that two different accounts cannot have the same index.
    Each index, starting from smallest (first) to largest(last) can be represented in a linear fashion, e.g.
    0 < 1 < 2 < 3 < 4 < ... < numAccounts
    This method of total ordering makes a circular wait impossible, since the last and largest number cannot be smaller than the first.
    This prevents the indices being able to be structured as a loop, so the deadlock cannot occur.
